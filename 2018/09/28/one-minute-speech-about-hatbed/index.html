one minute speech about hatbed

Hello everyone,

Our HATBED serves the IoT and WSN. It can be used to solve the problm of non-intrusive debugging. The architecture contains controller, observer and target. Computer server as controller which sends commands to observer. Observer collects debugging and tracing information and sends to controller. Target are MCUs which have Cortex-M3/M4 CPU. It provides the following three services.
The first is Network-Wide Remote Debugging, which includes global breakpoints, network-wide distributed assertions and automatic debugging.
The second is similar to UART printf and it named Flexible Software Tracing.
The third is Non-invasive Software Profiling, which can capture changes of interested data or peripherals, hitting of vertexes in control flows and CPU events. HATBED based on Hardware Assisted Tracing such as ITM/DWT and ETM. HATBED aims to use standard components to build a low-cost testbed.

Finally, thank you for listening, and we will work harder for HATBED.

Good morning, I will introduce Hardware Assisted Tracing Testbed for Embedded Networked Sensor Systems. We call it HATBED.
First is motivation, we need instruction-level tracing, no effect on execution, low overhead, low cost and easy to use in IoT and WSN debugging. We watched tracing technology based Hardware Assisted. For example, Cortex-M3/M4 CPU can achieve non-intrusive debugging with the help of DWT ( Data Watch Point and Trace ), ITM ( Instrumentation Trace Macrocell ) and ETM ( Embedded Trace Macrocell ). Meanwhile, we also noted modern MCUs usually have the core. So this is the basis of our research.
Here is system architecture. As the picture and table, we can see HATBED consists of targets, observers, and a controller. Targets are ARM Cortex-M3/M4 processors. An observer runs OpenOCD and GDB under Raspberry Pi 3B+, to debug targets through FT2232HL. And it captures tracing raw data of targets by using CY7C68013A based logic analyzer. Then the observer uploads the raw data to the controller which uses sigrok to decode the tracing raw data. All observers connect to the controller through WLAN/Ethernet. And controller sends debugging commands through TCP/IP protocol.
HATBED provides the following three services. The first is Network-Wide Remote Debugging, which includes global breakpoints, network-wide distributed assertions and automatic debugging. The second is similar to UART printf and it named Flexible Software Tracing. The third is Non-invasive Software Profiling. HATBED can capture changes of interested data or peripherals, hitting of vertexes in control flows and CPU events. It based on Hardware Assisted Tracing such as ITM/DWT and ETM.
And as the Compare showing us, HATBED is more flexible and efficient than other tracing schemes of testbeds.
We write scripts and MCU programs for preliminary evaluation. 72MHz CPU clock and 8MHz tracing clock. After testing. In code coverage rate under FreeRTOS. Before pre-analysis, ITM is not suitable for statement code coverage because of the limited tracing clock and bandwidth. But, ITM combined with DWT watchpoint could achieve the 90% function coverage rate. Here, we decode DWT watchpoint for interested task functions. ETM is much better than ITM. However, ETM is set to allow suspend CPU for capturing all execution information. In time overhead, ITM brings little overhead, and ETM brings a lot of overhead. There are two experiments can tell us ETM is not suitable for being always on. Turn it on when you need it. In power consumption, the experiment shows that ITM incurs about 1.53% increment and ETM results in about 1.8% reduction. The increase of time and the reduction of power are because that ETM slows down the CPU to prevent overflow of FIFO buffer. Decoding results show the captured ETM and ITM traces.
